# /product-init

初始化 `.claude/product/` 目录，通过对话创建产品文档。

## 核心理念

**Repository Is All You Need**

- 目标用户：**个人独立开发者**
- 痛点：产品思考散落在 Notion、Obsidian、Slack、Google Doc
- 解决：把**非技术层面**的产品决策集中到代码仓库

```
✅ 产品决策（.claude/product/）     ❌ 技术决策（CLAUDE.md）
"不做企业版"                        "不用 JavaScript"
"暂不支持日本市场"                   "用 PostgreSQL"
"先 ProductHunt 发布"               "用 React"
```

## 执行流程

### 1. 检查目录

```
.claude/product/ 已存在？
├─ 是 → 询问：补充缺失文件 / 重置 / 取消
└─ 否 → 继续
```

### 2. 引导式对话

**核心原则**：
- **一次一个问题** — 不要用多个问题压倒用户
- **优先多选题** — 比开放式问题更容易回答
- **每阶段确认** — 总结理解，确认后再进入下一阶段
- **YAGNI** — 无情地砍掉"可能需要"但"现在不需要"的功能

---

**阶段 1：核心（VISION.md）**

先问产品类型（多选）：
> "这个产品的核心是什么？"
> A. 解决一个具体痛点
> B. 提供娱乐/游戏体验
> C. 创造新的可能性/工具
> D. 其他

然后追问：
> "用一句话描述：这是什么？为什么做？"

**确认后进入下一阶段。**

---

**阶段 2：用户（USERS.md）**

提问：
> "谁是核心用户？描述一个具体的人"

如果用户回答模糊（如"所有人"），追问：
> "如果只能选一类人，最核心的是谁？"

**确认后进入下一阶段。**

---

**阶段 3：功能（SCOPE.md）**

**Step 1**：先问核心功能
> "产品最核心的 1-3 个功能是什么？"

**Step 2**：确认核心功能后，提供方案选择
> "关于产品范围，有几种常见做法：
> A. **精简版**：只做核心功能，快速验证（推荐）
> B. **完整版**：功能齐全，开发周期长
> C. **渐进式**：先 A 再逐步加功能
>
> 你倾向哪种？"

**Step 3**：如果用户选 B 或想加功能，应用 YAGNI
> "这个功能现在就需要吗？还是可以等验证后再加？"

**确认后进入下一阶段。**

---

**阶段 4：路线（ROADMAP.md）**

提问（多选）：
> "当前在什么阶段？"
> A. 想法验证（还没写代码）
> B. 开发中（正在做）
> C. 内测（小范围测试）
> D. 已上线（在运营）

然后追问：
> "下一个里程碑是什么？"

**确认后生成文档。**

---

### 3. 扩展文档（按需）

当用户提到以下关键词时，**主动提议**创建对应文件：

| 触发词 | 文件 | 示例 |
|-------|------|------|
| 价格、定价、收费、¥、$ | PRICING.md | "卖99块" |
| 发布、获客、渠道、推广、ProductHunt | MARKETING.md | "先在面包多上架" |
| 品牌、调性、风格、Logo、颜色 | BRAND.md | "想要专业感" |
| 运维、支持、客服、工单 | OPS.md | "用户怎么反馈问题" |
| 商业模式、盈利、变现 | BUSINESS.md | "靠增值服务赚钱" |

**行为**：
```
用户："核心版卖99块，体检报告免费试用"
AI："你提到了定价策略，要不要创建 PRICING.md 记录下来？"
```

**原则**：用户主动补充的细节 → 追加到对应文件，不强加框架

### 4. 更新 CLAUDE.md

- 不存在 → 创建
- 已存在 → 检查是否有 `## Product Context`
  - 没有 → 追加
  - 有 → 跳过

**推荐格式**（可根据项目情况调整）：

```markdown

## Product Context

本项目使用 `.claude/product/` 管理产品决策。

| 文档 | 内容 |
|------|------|
| VISION.md | [一句话核心差异] |
| USERS.md | [目标用户] |
| SCOPE.md | [核心功能] |
| ROADMAP.md | 阶段和决策记录 |

**ROADMAP 实时更新**：当完成功能、做出决策或改变方向时，主动提议更新。
```

**注意**：表格内容根据实际对话填写，不要用占位符。

## 参数支持

可以在命令后添加上下文：

```
/product-init
/product-init 参考 README.md
/product-init 这是一个 SaaS 产品，主要做...
```

## 文件结构

```
.claude/product/
├── ROADMAP.md      # ★ 核心：阶段、里程碑、Decision Log
├── VISION.md       # 为什么做
├── USERS.md        # 给谁做
├── SCOPE.md        # 做什么（最终形态）
└── [自由扩展]      # MARKETING.md, PRICING.md, BRAND.md, OPS.md...
```

## 引导原则

| 原则 | 说明 |
|-----|-----|
| 一次一个问题 | 不要用多个问题压倒用户 |
| 优先多选题 | 比开放式问题更容易回答 |
| 每阶段确认 | 总结理解，确认后再下一阶段 |
| YAGNI | 无情砍掉"可能需要"的功能 |
| 用户补充优先 | 用户说的就记录，不强加框架 |

## 常见误区处理

| 用户说 | 可能的实际意义 | 处理方式 |
|--------|---------------|----------|
| "目标用户是所有人" | 没想清楚 | 追问：最核心的一类用户是谁？ |
| "功能要简单" | 可能怕复杂 | 追问：简单是指少功能，还是易用？ |
| "不做 X" | 可能是暂时不做 | 追问：是永远不做，还是当前不做？ |
| "以后可能需要 Y" | 过度设计倾向 | 应用 YAGNI：现在需要吗？ |
| "竞品都有这个功能" | 跟风倾向 | 追问：你的用户真的需要吗？ |
