# Product Context Skill

## 核心理念

**Product Context 是产品/商业决策文档，不是技术文档。**

```
.product/  = 决策层（WHY、WHAT）  → 产品/商业决策
AGENTS.md  = 操作层（HOW）        → 技术实现细节
```

你是产品的"首席产品官"（CPO）。在编写任何代码之前，你必须理解：
- **为什么**要做这个功能（使命/愿景）
- **给谁**做（目标用户）
- **现在在哪**（当前阶段）

## 文件结构

```
.product/
├── VISION.md       # 为什么做（核心）
├── ROADMAP.md      # 时间轴 + 决策日志（核心）
├── USERS.md        # 给谁做（核心）
├── SCOPE.md        # 做什么/不做什么（核心）
├── BRAND.md        # 怎么说（扩展）
├── MARKETING.md    # 用户从哪来（扩展）
├── BUSINESS.md     # 钱从哪来（扩展）
└── OPS.md          # 怎么运行（扩展）
```

**4 + 4 结构**：
- **核心 4**：VISION + ROADMAP + USERS + SCOPE（必须有）
- **扩展 4**：BRAND + MARKETING + BUSINESS + OPS（按需添加）

**设计原则**：
- 扁平结构，一目了然
- 渐进式：先核心 4 个，按需添加扩展
- 每个文件都是决策，不是操作指南

## 命令

### /product init

在当前项目创建 `.product/` 目录结构（核心 4 个文件）。

**执行步骤**：
1. 检查是否已存在 `.product/` 目录
2. 如果已存在，询问用户是否要重置或补充
3. 如果不存在，创建目录和核心 4 个文件
4. 通过对话引导用户填写关键信息

**引导问题**：
- "用一句话描述你的产品解决什么问题？"
- "目标用户是谁？他们有什么痛点？"
- "这个产品与现有方案相比有什么优势？"
- "当前处于什么阶段？下一步要做什么？"
- "核心功能是什么？明确不做什么？"

**创建文件**（核心 4）：
```
.product/
├── VISION.md       # 使命、愿景、价值主张
├── ROADMAP.md      # 当前阶段、里程碑、决策日志
├── USERS.md        # 用户画像、痛点、场景
└── SCOPE.md        # 功能边界、非目标
```

**按需创建**（扩展 4）：
- BRAND.md - 当需要统一文案风格时
- MARKETING.md - 当开始考虑获客时
- BUSINESS.md - 当开始考虑商业化时
- OPS.md - 当需要做基础设施决策时

### /product save

智能分析当前对话，更新需要更新的产品文档。

**执行步骤**：
1. 分析本次会话的内容
2. 识别产品/商业决策（而非技术实现细节）
3. 判断哪些 `.product/` 文件需要更新
4. 提议更新内容，获得用户确认后执行
5. 更新 ROADMAP.md 的 Notes（如有重要决策）

**触发更新的内容**：
- 完成里程碑 → 更新 ROADMAP.md（勾选已完成项）
- 产品方向调整 → 更新 VISION.md
- 用户理解深化 → 更新 USERS.md
- 功能边界变化 → 更新 SCOPE.md
- 品牌调性确定 → 更新 BRAND.md
- 获客策略决定 → 更新 MARKETING.md
- 商业模式决定 → 更新 BUSINESS.md
- 基础设施决策 → 更新 OPS.md

**智能创建扩展文件**：
当讨论相关话题但文件不存在时，主动提议创建：
- 讨论文案风格 → "要创建 BRAND.md 吗？"
- 讨论获客渠道 → "要创建 MARKETING.md 吗？"
- 讨论定价/商业模式 → "要创建 BUSINESS.md 吗？"
- 讨论基础设施 → "要创建 OPS.md 吗？"

**不应该更新的内容**（放 AGENTS.md/docs）：
- 具体的技术实现方案
- 部署命令和配置
- API 设计和数据结构
- 编码规范

## 行为层规则

### 自动行为

当 `.product/` 目录存在时，在以下情况自动读取：

| 情况 | 读取文件 |
|------|----------|
| 会话开始 | ROADMAP.md（了解当前阶段） |
| 实现新功能前 | VISION.md + SCOPE.md（确保对齐，检查非目标） |
| 写用户可见文字时 | BRAND.md（保持调性一致，如存在） |
| 考虑用户需求时 | USERS.md（用户视角） |

### 半自动提示

在以下情况提示用户是否需要更新：

| 情况 | 提示 |
|------|------|
| 完成重要功能 | "需要更新 ROADMAP 吗？" |
| 做了产品决策 | "要记录到 Decision Log 吗？" |
| 讨论了用户场景 | "要更新 USERS.md 吗？" |
| 确定了新的边界 | "要更新 SCOPE.md 吗？" |
| 讨论扩展话题 | "要创建 [对应扩展文件] 吗？" |

## 文件职责速查

| 文件 | 回答的问题 | 决策内容 |
|------|-----------|----------|
| VISION.md | 为什么做？ | 使命、愿景、价值主张 |
| ROADMAP.md | 现在在哪？下一步？ | 阶段规划、里程碑、决策日志 |
| USERS.md | 给谁做？ | 用户画像、痛点、场景 |
| SCOPE.md | 做什么？不做什么？ | 功能边界、非目标 |
| BRAND.md | 怎么说话？ | 调性、风格、用词 |
| MARKETING.md | 用户从哪来？ | 获客渠道、推广策略 |
| BUSINESS.md | 钱从哪来？ | 定价、商业模式 |
| OPS.md | 怎么运行？ | 基础设施决策、成本结构 |

## 与 AGENTS.md / CLAUDE.md 的关系

`.product/` 和 `AGENTS.md`/`CLAUDE.md` 是**互补关系**：

| .product/ | AGENTS.md / CLAUDE.md |
|-----------|----------------------|
| 为什么写这个代码 | 怎么写代码 |
| 商业逻辑 | 技术规范 |
| 品牌调性 | 编码风格 |
| 产品愿景 | 构建命令 |
| 决策（选什么、为什么） | 操作（怎么做） |

## 注意事项

1. **决策 vs 操作** - `.product/` 只记录决策，操作放 AGENTS.md
2. **不要过度填充** - 保持简洁，信息够用即可
3. **版本控制** - 这些文件应该和代码一起提交
4. **AI 更新前确认** - 修改用户的产品文档前，先展示改动内容
